## Likely Interview Questions

### Architecture & Design

**Q: Why did you choose a multi-agent architecture instead of a single LLM chain?**
- Dental insurance has two fundamentally different data domains — unstructured benefit plan documents and structured claims data. A single chain would need to handle both, making prompts bloated and responses less accurate. Separate specialist agents let each one focus on its domain with tailored prompts, data access patterns, and response formats. The Router Agent acts as a lightweight triage layer using keyword heuristics first (fast, no API cost) and falling back to LLM classification only for ambiguous queries.

**Q: Walk me through the design patterns you used and why.**
- **Factory Pattern** — `LLMProviderFactory` and `EmbeddingProviderFactory` abstract away model creation so swapping from GPT-4o-mini to another model is a one-line config change, not a code rewrite.
- **Strategy Pattern** — `BaseAgent` defines a common interface; `BenefitsAgent` and `ClaimsAgent` are interchangeable strategies selected at runtime based on intent classification.
- **Repository Pattern** — `ClaimsRepository` wraps all data access to the claims CSV, keeping the service layer decoupled from the storage format.
- **Dependency Injection** — FastAPI's `Depends()` wires all services together, making unit testing trivial since you can inject mocks.
- **Chain of Responsibility** — The Router classifies intent, then delegates to the appropriate specialist agent, forming a processing chain.

**Q: How would you scale this to production?**
- Replace ChromaDB with a managed vector database (Pinecone, Weaviate, or pgvector on RDS). Swap the in-memory conversation store for Redis. Add authentication/authorization middleware. Deploy the FastAPI backend as a containerized service behind a load balancer. Use async streaming (SSE) for real-time chat responses. Add observability with OpenTelemetry tracing on every LLM call.

### RAG Pipeline

**Q: How does your RAG pipeline work?**
- Documents (benefit plans, FAQs, CDT procedure codes) are loaded, split into ~1024-character chunks with 200-character overlap using LangChain's `RecursiveCharacterTextSplitter`, embedded with OpenAI's `text-embedding-3-small`, and stored in ChromaDB. At query time, the retrieval service searches across all collections, ranks results by cosine distance, takes the top 8 chunks, prefixes each with its source file name, and injects them into the system prompt as context for GPT-4o-mini to generate a grounded answer.

**Q: Why 1024 chunk size and 200 overlap?**
- Initially I used 512/50 and found that specific questions like "Does the PPO Gold plan cover root canals?" returned "I don't have information" because the root canal coverage section was split across chunk boundaries. Increasing to 1024/200 keeps full coverage sections intact within a single chunk. The 200-character overlap ensures no information is lost at boundaries.

**Q: How do you prevent hallucinations in the RAG responses?**
- Three layers: (1) The system prompt explicitly says "Answer ONLY based on the context provided — never fabricate plan details" and instructs the model to say "I don't have information" when context is insufficient. (2) Each context chunk is labeled with `[Source: filename]` so the LLM can attribute information to specific plans. (3) The evaluation pipeline includes an LLM-as-judge faithfulness check and a dedicated hallucination detector that flags unsupported claims.

**Q: Why did you add source file labels to context chunks?**
- I discovered that when a chunk contained "Basic Services (80% after deductible): D3310 Root canal therapy — Covered" but didn't mention "PPO Gold" anywhere in the chunk text, the LLM couldn't confidently attribute the coverage to a specific plan. Adding `[Source: delta_ppo_gold.md]` before each chunk solved this — it went from "I don't have information" to a correct, cited answer.

### Evaluation & Responsible AI

**Q: How do you evaluate your RAG system's quality?**
- Four metrics: (1) **Faithfulness** — LLM-as-judge checks if each factual claim in the response is supported by the source context. (2) **Relevance** — scores how directly the response addresses the user's question on a 0-1 scale. (3) **Hallucination detection** — identifies unsupported claims. (4) **Responsible AI checks** — regex-based detection for PII leakage (SSN, phone, email), medical advice, demographic bias, and scope violations. There's also a batch evaluation mode that runs all metrics against a golden Q&A dataset.

**Q: What responsible AI concerns exist in dental insurance AI?**
- PII leakage is the biggest risk — member IDs, SSNs, and personal health information could appear in responses. Medical advice is another — the system must explain insurance coverage without recommending treatments. Bias could appear if the model makes assumptions about coverage based on demographics. My system has automated checks for all of these, plus the prompt explicitly states "Never provide medical advice — you can only explain insurance coverage."

**Q: What's your faithfulness threshold and why?**
- 0.8 (80%). Below that, responses have too many claims not grounded in source documents. In insurance, giving a member wrong coverage information could lead to unexpected bills, so the bar for accuracy needs to be high. The system flags responses below this threshold for review.

### Claims Analysis

**Q: How does the anomaly detection work?**
- For each procedure code, the system calculates the mean and standard deviation of billed amounts across all claims. Any claim where the billed amount exceeds 2.5 standard deviations above the mean is flagged as anomalous. The synthetic data generator intentionally seeds ~5% of claims with inflated amounts (2-5x typical), simulating real-world billing irregularities. The system detected 51 out of 1,200 (4.25%), which closely matches the seeded anomaly rate.

**Q: Why synthetic data instead of real claims?**
- Real dental claims contain PHI (Protected Health Information) regulated by HIPAA. Using synthetic data lets me demonstrate the full pipeline — statistical analysis, anomaly detection, category breakdowns — without any compliance risk. The generator creates realistic distributions: proper CDT codes, typical billing ranges per procedure, multiple claim statuses, and a realistic ~67% approval rate.

### Technical Implementation

**Q: Why FastAPI over Flask or Django?**
- FastAPI gives native async support (critical for LLM API calls that can take seconds), automatic OpenAPI documentation, built-in request validation with Pydantic, and dependency injection via `Depends()`. Django is overkill for an API-only service, and Flask lacks the async-first design and type validation.

**Q: Why ChromaDB for the vector store?**
- For a portfolio project, ChromaDB runs locally with zero infrastructure — no database server, no cloud account needed. It supports persistent storage, integrates directly with LangChain, and handles the scale of this project (64 document chunks) easily. In production I'd evaluate Pinecone or pgvector for managed scaling and backup.

**Q: Why GPT-4o-mini instead of GPT-4o or GPT-4?**
- Cost and latency. GPT-4o-mini is ~60x cheaper than GPT-4 and significantly faster, while still being capable enough for the classification, Q&A, and evaluation tasks in this project. For a production system handling thousands of queries per day, the cost difference is substantial. The architecture makes it easy to swap models via config if higher capability is needed.

**Q: How do you handle conversation memory?**
- A `BufferWindowStrategy` keeps the last N messages (configurable, default 10) in the conversation history. This is passed to the LLM as part of the system prompt context so it can reference prior exchanges. The conversation manager also tracks conversation IDs so multiple users can have independent sessions. For production, I'd swap the in-memory store for Redis with TTL-based expiration.

**Q: How did you test this project?**
- 57 tests across three layers: **Unit tests** mock all external dependencies (LLM, vector store) and verify domain models, agent routing logic, evaluation metrics, and prompt templates. **Integration tests** verify the RAG pipeline end-to-end and API endpoint behavior. Tests use `pytest` with fixtures defined in `conftest.py` for shared mock objects. All tests run in ~1 second since external calls are mocked.

### Behavioral / Situational

**Q: What was the hardest bug you encountered?**
- The RAG pipeline retrieved the correct source documents but the LLM still said "I don't have information" for specific questions like "Does the PPO Gold plan cover root canals?" Debugging showed the root canal coverage chunk was being retrieved (8th out of 8 results) and included in the context, but the chunk didn't mention "PPO Gold" anywhere — just "Basic Services (80% after deductible)." The LLM couldn't attribute the coverage to a specific plan. The fix was adding `[Source: delta_ppo_gold.md]` metadata labels to each chunk in the context, which immediately solved it.

**Q: If you had more time, what would you add?**
- Streaming chat responses via Server-Sent Events for better UX. A feedback loop where users rate answers and low-rated responses get flagged for prompt tuning. Integration with a real CDT code database API. Cost tracking per conversation to monitor LLM spend. An admin dashboard showing aggregate evaluation metrics over time. And containerization with Docker Compose for one-command deployment.

**Q: Why is this project relevant to Delta Dental specifically?**
- It uses Delta Dental's actual business domain — dental benefit plans with real CDT procedure codes, claims processing workflows, and the specific plan types Delta Dental offers (PPO, HMO). The knowledge base includes real Delta Dental programs sourced from deltadentalmi.com: DeltaVision, Special Health Care Needs benefit, Medicare Advantage partners in Michigan, Healthy Kids Dental, and LifeSmile Wellness. The multi-agent architecture mirrors how a real insurance company would handle member queries: routing to benefits specialists vs. claims analysts. The responsible AI guardrails (PII protection, no medical advice) address real compliance requirements in the insurance industry. And the tech stack (Python, LangChain, OpenAI, FastAPI) matches what the job posting asks for.

**Q: What Delta Dental programs does your system know about?**

This gives members two layers of cost savings and balance-billing protection. No other carrier offers this dual-network structure. My system can explain this when members ask about plan differences.
